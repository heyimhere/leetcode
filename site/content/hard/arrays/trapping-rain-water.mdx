## Problem

Given `n` non-negative integers representing an elevation map where each bar has width 1, compute how much water it can trap after raining.

**Example:**
```
Input:  height = [0,1,0,2,1,0,1,3,2,1,2,1]
Output: 6

     ■
  ■~~■~■~~■~■
■~■~~■~■~~■~■
[0,1,0,2,1,0,1,3,2,1,2,1]
~~ = trapped water
```

---

## How to Think About It

**Brute force:** For each index, find the max height to its left and max height to its right. Water at that index = `min(leftMax, rightMax) - height[i]`. This is O(n²) because you scan left/right for every position.

**Precompute left/right arrays:** Build a `leftMax[]` and `rightMax[]` array first, then compute water in one pass. O(n) time, O(n) space. This is the stepping stone toward the optimal.

**Two-pointer (optimal):** You don't need both arrays simultaneously. The key observation:

> Water at any position is bounded by `min(leftMax, rightMax)`. If `height[left] < height[right]`, then `leftMax` is the bottleneck — you know the right side has a taller wall somewhere. Process the left side and move `left++`. Otherwise, process the right side and move `right--`.

You maintain `leftMax` and `rightMax` as you go, never needing the full arrays. This gets you O(1) space.

**The insight in one sentence:** Whichever side has the shorter current boundary is the one you can safely process — because the opposite side is guaranteed to be at least as tall.

## Big-O Breakdown

| | Complexity | Reasoning |
|---|---|---|
| **Two pointers** | Time O(n) | Single pass, no arrays |
| | Space O(1) | Four variables: left, right, leftMax, rightMax |
| **Precomputed arrays** | Time O(n) | Three passes |
| | Space O(n) | Two arrays of length n |
| **Brute force** | Time O(n²) | Nested loops |
| | Space O(1) | No extra storage |

This problem is a great example of how thinking about **what information you actually need at each step** (just the running max, not the full arrays) unlocks a space optimization.
